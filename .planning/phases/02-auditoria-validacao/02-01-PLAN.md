<task type="auto">
  <name>Task 1: Auditar regras de validação frontend (client-side)</name>
  <files>apps/dashboard/src/lib/poker/validation.ts</files>
  <action>
Ler e analisar completamente o arquivo de validação frontend (~1,200 linhas).

Para cada regra de validação identificada, documentar:
1. **Nome da regra** - identificador único
2. **Descrição** - o que a regra valida
3. **Implementação** - como está implementada (código resumido)
4. **Inputs** - dados necessários para validação
5. **Outputs** - resultado (error, warning, info)
6. **Edge cases** - casos extremos considerados ou não
7. **Dependências** - outras regras ou dados necessários
8. **Severidade** - blocking vs warning

Categorizar regras por tipo:
- Structure validation (tabs, columns)
- Player validation (IDs, duplicates)
- Transaction validation (balance, types)
- Session validation (rake, totals)
- Period validation (dates, ranges)
- Hierarchy validation (agents, cycles)

Identificar:
- Regras que faltam validação de edge cases
- Regras com lógica potencialmente incorreta
- Regras que deveriam ser blocking mas são warning
- Mensagens de erro que não ajudam o usuário
  </action>
  <verify>Documentação contém análise de todas as regras encontradas no arquivo validation.ts</verify>
  <done>Todas as regras de validação frontend documentadas e analisadas</done>
</task>

<task type="auto">
  <name>Task 2: Auditar validação backend e comparar com frontend</name>
  <files>apps/api/src/trpc/routers/poker/imports.ts, apps/api/src/schemas/poker/imports.ts</files>
  <action>
Ler e analisar os arquivos de importação backend (imports.ts ~1,667 linhas + schemas).

1. **Identificar todas as validações backend:**
   - Validações em Zod schemas (import input validation)
   - Validações em procedures (validate, process)
   - Validações implícitas (database constraints)

2. **Criar matriz de paridade frontend-backend:**

| Regra | Frontend | Backend | Paridade | Gap |
|-------|----------|---------|----------|-----|
| Structure | ✓/✗ | ✓/✗ | Match/Diff | Descrição |

3. **Analisar cada diferença:**
   - Por que existe diferença?
   - Qual comportamento está correto?
   - Risco se dados inválidos passarem

4. **Verificar ordem de execução:**
   - Frontend valida primeiro (preview)
   - Backend valida depois (process)
   - Há validações que só acontecem em um lado?

5. **Documentar validações implícitas:**
   - Database constraints (NOT NULL, UNIQUE, FK)
   - TypeScript types (compile-time only)
   - RLS policies
  </action>
  <verify>Matriz de paridade completa com todas as regras frontend e backend mapeadas</verify>
  <done>Comparação frontend-backend documentada com gaps identificados</done>
</task>
