## Executive Summary

This audit examines the rake calculation and distribution system in Mid Poker, focusing on mathematical accuracy, hierarchy flow, and settlement calculations.

### Key Findings

| Area | Status | Risk Level |
|------|--------|------------|
| Rake Source Data | Working | LOW |
| Rake Aggregation | Working | LOW |
| Rakeback Calculation | Partially Working | MEDIUM |
| Hierarchy Distribution | Incomplete | HIGH |
| Settlement Calculations | Flawed | CRITICAL |

### Critical Issues Identified

1. **Settlement uses chip_balance, not rake** - The `closeWeek` procedure calculates settlements from `chip_balance`, not from actual rake generated during the period
2. **Rakeback calculated at settlement, not accumulation** - Rakeback is calculated on gross settlement amount, not on rake generated
3. **No rake-based commission system** - Agents are paid rakeback on player balance, not commission on rake generated by their players
4. **Inconsistent rakeback_percent field** - Schema uses `rakeback_percent` but settlements reference `rakeback_percentage`

## 1. Rake Sources and Storage

### 1.1 Where Rake is Generated

Rake originates from game sessions and is stored at multiple levels:

#### poker_session_players (Partidas level)
```
Fields:
- rake: numeric(14,2) - Total rake from this player's session
- rake_ppst: numeric(14,2) - PPST rake component
- rake_ppsr: numeric(14,2) - PPSR rake component
- winnings: numeric(14,2) - Player's session result
```

#### poker_player_summary (Geral level - aggregated)
```
Fields:
- rake_total: numeric(14,2) - Total rake for period
- rake_ppst: numeric(14,2) - PPST rake for period
- rake_non_ppst: numeric(14,2) - Non-PPST rake
- rake_ppsr: numeric(14,2) - PPSR rake for period
- club_earnings_*: Various game type earnings
- winnings_total: numeric(14,2) - Total player winnings
```

### 1.2 Data Flow

```
PPPoker Spreadsheet
    |
    v
Import Process (imports.ts)
    |
    +-> poker_session_players (per session per player)
    |
    +-> poker_player_summary (aggregated per period)
    |
    v
Dashboard Analytics (analytics.ts)
    |
    v
Settlement Calculation (settlements.ts)
```

## 3. Hierarchy Distribution Analysis

### 3.1 Player Hierarchy Schema

From `poker_players` table:
```
poker_players
  - id (PK)
  - type: 'player' | 'agent'
  - agent_id -> poker_players.id (self-referential FK)
  - super_agent_id -> poker_players.id (self-referential FK)
  - rakeback_percent: numeric(5,2)
```

### 3.2 Hierarchy Relationships

```
Super Agent (type='agent', super_agent_id=NULL)
    |
    +-> Agent (type='agent', super_agent_id=SuperAgent.id)
    |      |
    |      +-> Player (type='player', agent_id=Agent.id)
    |      +-> Player (type='player', agent_id=Agent.id)
    |
    +-> Agent (type='agent', super_agent_id=SuperAgent.id)
           |
           +-> Player (type='player', agent_id=Agent.id)
```

### 3.3 Current Implementation Issues

#### Issue 1: Rakeback Only at Agent Level
The `rakeback_percent` field is on `poker_players`, but only agents have meaningful values. Players themselves don't have individual rakeback percentages - they inherit from their agent.

**Code Evidence (analytics.ts line 442-448):**
```typescript
const { data: agentsData } = await supabase
  .from("poker_players")
  .select("id, rakeback_percent")
  .eq("team_id", teamId)
  .eq("type", "agent");  // Only agents queried for rakeback
```

#### Issue 2: Super-Agent Commission Not Implemented
Super-agents have a `super_agent_id` reference, but there's no commission cascade:
- Super-agent doesn't automatically receive a cut of agent's rakeback
- No tiered commission structure implemented

**Missing Logic:**
```
Expected: Super-Agent receives X% of Agent's commission
Actual: Super-Agent only receives rakeback on their OWN managed players
```

#### Issue 3: Players Without Agents Have No Rakeback
Players without an assigned agent (`agent_id = null`) generate rake but no rakeback is calculated for them.

**This is likely INTENDED behavior** - only players under agents get rakeback deals.

## 4. Settlement System Analysis

### 4.1 closeWeek Procedure

**File:** `settlements.ts` lines 377-482

**Current Implementation:**
```typescript
// Get players with non-zero chip_balance
const { data: players } = await supabase
  .from("poker_players")
  .select("id, nickname, chip_balance, agent_id, rakeback_percentage")
  .eq("team_id", teamId)
  .eq("status", "active")
  .neq("chip_balance", 0);

// Create settlement per player
const settlements = players.map((player) => {
  const grossAmount = player.chip_balance ?? 0;
  const rakebackAmount = grossAmount > 0
    ? (grossAmount * (player.rakeback_percentage ?? 0)) / 100
    : 0;
  const netAmount = grossAmount - rakebackAmount;
  // ...
});
```

### 4.2 CRITICAL ISSUES IDENTIFIED

#### Issue C1: Settlement Based on chip_balance, Not Rake

**Current:** `grossAmount = player.chip_balance`
**Expected:** `grossAmount = SUM(player's rake for period)`

The `chip_balance` represents the player's current balance (what they owe or are owed), NOT the rake they generated. These are fundamentally different:

- **chip_balance:** Result of chip movements (bought/sold)
- **rake:** Fee extracted from gameplay

**Impact:** Settlements are calculated on the wrong metric entirely.

#### Issue C2: Rakeback Formula Applied to Balance, Not Rake

**Current:** `rakebackAmount = chip_balance * rakeback_percentage / 100`
**Expected:** `rakebackAmount = rake_generated * rakeback_percentage / 100`

Applying rakeback percentage to chip balance makes no business sense:
- A player who bought $1000 in chips but lost $800 would have $200 balance
- Current: rakeback = $200 * 10% = $20
- Expected: rakeback should be based on rake paid, not balance

#### Issue C3: Field Name Mismatch

**Schema:** `poker_players.rakeback_percent`
**Settlement Query:** `rakeback_percentage` (undefined field)

```typescript
// settlements.ts line 400
.select("id, nickname, chip_balance, agent_id, rakeback_percentage")
//                                             ^^^^^^^^^^^^^^^^^^^
// Should be: rakeback_percent
```

This means `player.rakeback_percentage` is always `undefined`, and rakeback is always 0!

#### Issue C4: No Period-Based Rake Aggregation

The `closeWeek` procedure should:
1. Query `poker_player_summary` for the period
2. Sum rake per player for that period
3. Calculate settlement based on period rake

**Current implementation completely ignores period-based rake data.**

#### Issue C5: Agent Settlements Not Created

The current implementation creates settlements for players, but doesn't create commission settlements for agents based on their players' rake.

**Expected Flow:**
1. Calculate player settlement (rake-based)
2. Calculate agent commission (% of player rake)
3. Create both player and agent settlements

### 4.3 Settlement Schema Analysis

```sql
poker_settlements
  - gross_amount: numeric(14,2) -- Should be rake-based
  - rakeback_amount: numeric(14,2) -- Currently 0 due to field mismatch
  - commission_amount: numeric(14,2) -- Always 0, not implemented
```

**Rakeback Percent Storage:**
The schema includes `rakeback_percent_used` but it's never populated in the current implementation.

## 5. Rakeback System Deep Dive

### 5.1 How Rakeback Should Work

```
Player plays game -> Rake extracted -> Player earns rakeback on rake paid

Example:
- Player pays $100 in rake
- Player's agent has 30% rakeback deal
- Player receives $30 rakeback at settlement
- Club retains $70
```

### 5.2 Current Implementation

```
Dashboard Analytics (CORRECT):
- Calculates rakeback = agent_rake * agent_rakeback_percent / 100
- Shows correct total rakeback in dashboard

Settlement (INCORRECT):
- Calculates rakeback = chip_balance * (undefined) / 100
- Always results in 0 rakeback due to field mismatch
```

### 5.3 Who Defines Rakeback Percentage

**Current:** Set per agent in `poker_players.rakeback_percent`
**Usage:** Applied to all players under that agent

This is a simplistic model - all players under an agent get the same rakeback %. Some systems allow per-player rakeback negotiation.

### 5.4 When Rakeback is Paid

**Expected:** At weekly settlement (`closeWeek`)
**Actual:** Never paid due to calculation errors

### C. Settlement Formula (Correct vs Current)

**Current (INCORRECT):**
```
grossAmount = chip_balance
rakebackAmount = chip_balance * rakeback_percentage / 100  // rakeback_percentage undefined!
netAmount = grossAmount - rakebackAmount
```

**Correct:**
```
grossAmount = chip_balance (what player owes/is owed)
rakeGenerated = SUM(poker_player_summary.rake_total) for period
rakebackAmount = rakeGenerated * agent.rakeback_percent / 100
netAmount = grossAmount - rakebackAmount (balance minus rakeback credit)
```

Or alternatively:
```
grossAmount = rakeGenerated (rake-based settlement)
rakebackAmount = grossAmount * agent.rakeback_percent / 100
netAmount = grossAmount - rakebackAmount
```

The choice depends on business requirements - whether settlements are balance-based (what to pay/collect) or rake-based (commission calculation).

## 10. Conclusion

The rake calculation system in the analytics dashboard is **mathematically correct**. The data sources are appropriate, the aggregations are accurate, and the formulas follow standard poker room accounting.

However, the **settlement system is critically flawed**:
1. Uses the wrong data source (chip_balance vs rake)
2. Has a field name bug causing rakeback to always be 0
3. Doesn't aggregate period-based rake for calculations
4. Doesn't create agent commission settlements

These issues mean that while the dashboard shows correct information, the actual weekly settlement process produces incorrect results and fails to pay out rakeback.

**Immediate action required:** Fix the settlement calculation before any production weekly closings to prevent financial discrepancies.
